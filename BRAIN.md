## Networking & Communication

**Client-Server Architecture** -- The foundation of web applications. A client (browser, mobile app) sends requests to a server, which processes them and returns responses. The server runs continuously, waiting for incoming connections. The client locates the server via its IP address, resolved through DNS.

**IP Addresses & DNS** -- Every publicly deployed server has a unique IP address. DNS (Domain Name System) maps human-readable domain names to IP addresses. When you type a URL, your browser queries a DNS server for the corresponding IP, then connects to that address. IPv4 uses 32-bit addresses (~4 billion), IPv6 uses 128-bit for vastly more. DNS records include A records (domain -> IPv4) and AAAA records (domain -> IPv6). DNS is overseen by ICANN; registrars like Namecheap sell domain names. DNS mappings are cached at browser, OS, ISP, and replicated DNS servers globally. Updating DNS records can take hours or even days because of global replication. [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] covers DNS caching at multiple stages and geo-replication in detail.

**DNS at Scale (caching + provider round-robin)** -- A third-party DNS provider has rate limits, so DNS resolution becomes a real bottleneck for systems that touch many domains. Two practical mitigations: cache domain-to-IP mappings in Redis so you hit the provider once per domain and serve the rest from cache, and round-robin across multiple DNS providers to spread load and survive any single provider rate-limiting or going down. The [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] design treats DNS as a commonly overlooked bottleneck and praises provider round-robin as a sign you think like an engineer solving a live problem rather than reciting a textbook.

**HTTP/HTTPS** -- HTTP is the request-response protocol for web communication. Requests include headers (metadata, cookies) and optionally a body (form data). Responses carry status codes: 2xx success, 3xx redirect, 4xx client error, 5xx server error. Common methods: GET (read), POST (create), PUT/PATCH (update), DELETE (remove). HTTP sends data in plain text; HTTPS encrypts it via SSL/TLS. HTTP is stateless: each request is independent, containing all necessary information.

**TCP & UDP** -- TCP (transport layer) ensures reliable, ordered delivery via sequence numbers and a three-way handshake. Each packet has a TCP header with port numbers and control flags. UDP is faster but unreliable: no connection setup, no delivery guarantee. UDP is preferred for time-sensitive applications (video calls, live streaming) where speed matters more than occasional data loss.

**Data Packets** -- When computers communicate over a network, they exchange packets. Each packet contains an IP header (sender/receiver addresses) and application-layer data formatted per the protocol (HTTP, etc.). The Internet Protocol governs how packets are sent and received.

**WebSockets** -- Enable persistent, bidirectional communication between client and server over a single connection. The client initiates a WebSocket handshake, and once established, either side can send messages at any time. Eliminates the need for polling. Used for chat apps, live dashboards, multiplayer games, stock tickers, collaborative editors. [[problems/Uber/Uber|Uber]] uses WebSockets for real-time ride tracking. [[problems/WhatsApp/WhatsApp|WhatsApp]] uses them as the core transport (real WhatsApp uses plain TLS to skip WebSocket overhead, same principle: a persistent connection ferrying messages both ways). [[problems/Google Docs/Google Docs|Google Docs]] uses one WebSocket per open doc for editing. REST cannot push server-to-client, which is why these systems abandon request/response. For scaling WebSocket connections across many service instances, combine with Redis Pub/Sub: publishers write to a channel, and whichever instance holds the recipient's socket receives and forwards.

**Real-Time Transport Decision Tree** -- A reusable rubric for choosing a real-time channel, from [[sources/Design WhatsApp w an Ex-Meta Senior Manager|WhatsApp]]: if it is not latency sensitive, use **long polling**; if the server only pushes to the client (no frequent client-to-server), use **SSE**; if there is frequent bidirectional traffic, use **WebSockets**; if it is peer-to-peer audio/video, use **WebRTC**. WhatsApp is latency sensitive, bidirectional, no A/V in scope, so WebSockets win. [[problems/Google Docs/Google Docs|Google Docs]] applies the same logic: SSE loses because the client also sends ops, long polling loses on per-message handshake overhead.

**Server-Sent Events (SSE)** -- A persistent unidirectional connection (server -> client only). Lighter than WebSockets when you only need the server to push updates to the client. Used for real-time seat maps, live feeds, notifications. [[problems/Ticketmaster/Ticketmaster|Ticketmaster]] uses SSE for real-time seat availability updates. Choose SSE over WebSockets when communication is strictly one-way (server to client).

**Long Polling** -- The client sends an HTTP request that the server holds open (30-60 seconds) until it has data to return. The client immediately re-sends. Simplest real-time approach, requires no extra infrastructure. Best when users are on the page briefly (1-5 minutes), or when a response is expected within a short window. A poor fit when changes arrive unpredictably over a long span (the [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] sync case), where adaptive polling is preferred.

**Webhooks** -- Allow a server to notify another server when an event occurs via HTTP POST. The receiver registers a callback URL with the provider. When the event fires, the provider sends data to that URL. Eliminates polling between servers. Example: Stripe sends a webhook to your booking service when a payment succeeds.

**Proxy Servers** -- Act as intermediaries between clients and servers. A **forward proxy** sits in front of clients, hiding their IP and controlling outbound access (use cases: internet monitoring in organizations, caching, anonymizing web access). A **reverse proxy** sits in front of servers, handling load balancing, SSL termination, caching, and security. Reverse proxies hide backend server topology from clients. Application servers behind a reverse proxy can be in a private network, disconnected from internet. [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] covers forward/reverse proxy types in depth: open, transparent, anonymous, distorting, and high-anonymity (elite) proxies.

**Firewalls & Ports** -- Firewalls monitor and control network traffic. Ports identify specific processes on a device; combined with an IP address, they create a unique network service identifier. Reserved ports: 80 (HTTP), 443 (HTTPS), 22 (SSH).

## APIs & Protocols

**REST API** -- The most widely used API style and the safe default in interviews. Stateless (each request is self-contained), resource-oriented (users, orders, products), uses standard HTTP methods. Simple, scalable, easy to cache. Downsides: can overfetch or underfetch data, may need multiple requests for related resources. Uses JSON for data exchange. Going non-standard (gRPC, raw function signatures) without a reason risks yellow flags from REST-purist interviewers, per [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]].

**GraphQL** -- Introduced by Facebook in 2015. Clients request exactly the fields they need in a single query, avoiding overfetch/underfetch. Strongly typed. All requests are POST. Returns HTTP 200 even for errors (details in response body). Trade-offs: more server-side processing, harder to cache than REST.

**gRPC** -- Google's RPC framework built on HTTP/2. Uses Protocol Buffers for serialization (efficient but less human-readable than JSON). Supports multiplexing and server push. Excellent for microservice-to-microservice communication where bandwidth efficiency matters. Requires HTTP/2 support.

**RPC (Remote Procedure Call)** -- A protocol allowing a program to execute code on a remote machine as if it were local. Abstracts network communication details. Many application-layer protocols use RPC internally (HTTP requests can trigger backend RPC calls, SMTP servers use RPC for database interaction).

**WebSocket "API" (commands, not REST)** -- There is no standard for a WebSocket API. Model it as two command sets: commands sent (client -> server) and commands received (server -> client). [[problems/WhatsApp/WhatsApp|WhatsApp]] uses sent commands like createChat, sendMessage, createAttachment, modifyParticipants, ack, and received commands like newMessage, chatCreated, participantsChanged. [[problems/Google Docs/Google Docs|Google Docs]] frames its live channel as framed messages: op (an edit tagged with the baseRevision it was authored against), ack, cursor, presence. Flag explicitly that this scaffolding is iterative, so the interviewer does not think you missed the standard.

**Internal System Interface (not a public API)** -- For backend components the client never calls directly, model the interface as an internal function (RPC or in-process call), not a REST endpoint. The [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Distributed Rate Limiter]] is modeled as `isAllowed(clientId, rule) -> { allowed, remaining, resetAt }`. The [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] is described as inputs (seed URLs) and outputs (stored text), not endpoints.

**API Gateway** -- A centralized entry point for all client requests in a microservices architecture. Handles routing to the correct service, authentication, rate limiting, SSL termination, logging, and monitoring. Examples: AWS API Gateway (managed, auto-scaling). Simplifies client interaction since they hit one endpoint instead of multiple services. Note its limits: AWS managed API Gateway caps POST body size at 10 MB, which is why large-file uploads ([[problems/WhatsApp/WhatsApp|WhatsApp]] media, [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] 50 GB files) must bypass it via pre-signed URLs.

**API Best Practices** -- Design endpoints reflecting resource relationships (`/users/:userId/orders`). Use pagination (limit/offset or cursor-based). GET requests must be idempotent and never mutate data. Maintain backward compatibility with versioning (`/v2/products`). Set rate limits to prevent DDoS. Configure CORS to control which domains can access your API. [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] emphasizes: always start with RDBMS-based schema in interviews (jumping to NoSQL is a red flag), and never name a technology (Redis, Elasticsearch) without explaining the architectural decisions behind it.

**API Security: identity from the header, never the body** -- User identity must come from the JWT or session token in the request header, never from the request body. Putting userId in the body lets anyone falsify a request and act on another user's behalf. Cited across [[problems/Ticketmaster/Ticketmaster|Ticketmaster]] (two-phase booking), [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]], and [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Rate Limiter]] (edge placement loses business context, so encode needed claims like premium status in a signed JWT).

**Cursor-Based Pagination** -- For feeds with rapidly changing data, use cursor-based pagination instead of offset-based. With offset, new items inserted at the top shift all offsets, causing users to see duplicate or skipped items. Cursor pagination uses a stable reference point (e.g., a post ID or timestamp) so the next page always starts right after the last item seen, regardless of insertions. [[problems/Twitter Feed/Twitter Feed|Twitter Feed]] and [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]] both page an append-at-head feed with a timestamp cursor (`createdAt < cursor`) for gap-free, duplicate-free infinite scroll.

**Error Responses & Rate-Limit Headers** -- Do not return an opaque 500 when rejecting. The [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Distributed Rate Limiter]] returns HTTP 429 (Too Many Requests) plus headers: `X-RateLimit-Limit` (the ceiling), `X-RateLimit-Remaining` (requests left this window), `X-RateLimit-Reset` (when it resets), and often `Retry-After`. Interviewers expect you to know 429; the exact header names matter less than communicating to the client what happened and when to retry.

**Email Protocols** -- SMTP for sending email between servers. IMAP for retrieving email while keeping it on the server (multi-device access). POP3 for downloading email to a single local client.

**WebRTC** -- Enables browser-to-browser communication (voice, video, file sharing) without plugins. Essential for video conferencing and live streaming. The peer-to-peer / audio-video branch of the real-time transport decision tree.

**MQTT & AMQP** -- MQTT is lightweight, ideal for IoT devices with limited processing power and low bandwidth. AMQP provides robust, secure message-oriented middleware for enterprise systems (used by RabbitMQ).

## Data Storage

**SQL Databases** -- Store data in tables with strict schemas and relationships (foreign keys). Follow ACID properties: Atomicity (all-or-nothing transactions), Consistency (valid state after each transaction), Isolation (concurrent transactions don't interfere), Durability (committed data persists). Ideal for structured data requiring strong consistency: banking, financial systems, booking platforms. Examples: PostgreSQL, MySQL, SQLite. The "relational can't scale" belief is wrong: Shopify reported MySQL doing millions of transactions/sec on Black Friday ([[sources/Design WhatsApp w an Ex-Meta Senior Manager|WhatsApp]]). [[problems/Google Docs/Google Docs|Google Docs]] uses Postgres for its append-only op log precisely because a unique constraint enforces ordering cleanly.

**NoSQL Databases** -- Flexible, schema-less databases optimized for scalability and performance. Types: key-value (Redis), document (MongoDB), graph (Neo4j), wide-column (Cassandra, DynamoDB). Trade off strict consistency for horizontal scalability. Good for unstructured data, rapid iteration, simple queries. Many modern NoSQL databases now support ACID (e.g., DynamoDB transactions). The SQL vs NoSQL debate is less relevant than identifying the specific database qualities you need, and several sources call it an outdated, uninteresting debate (modern engines converge). [[System Design Interview - Design Ticketmaster w a Ex-Meta Staff Engineer|Ticketmaster interview]], [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]], and [[problems/WhatsApp/WhatsApp|WhatsApp]] all stress required qualities (write throughput, keyed lookups, single-digit-ms latency, atomic transactions) over the label.

**Wide-Column Stores & DynamoDB Modeling** -- Wide-column stores (DynamoDB, Cassandra) model relationships with a partition key plus a sort key, and a Global Secondary Index (GSI) for the reverse access pattern (the GSI behaves like a second table the engine maintains internally and consistently). [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]] models the Follow table with partition=userFollowing/sort=userFollowed and a GSI reversing the keys to answer "who follows X." [[problems/WhatsApp/WhatsApp|WhatsApp]] keys ChatParticipant on chatId with a participantId GSI for "which chats is this user in." Two hard constraints to know: a DynamoDB range query caps at 1 MB returned (~100k small entries, then page), and a DynamoDB transaction caps at 100 records, which bounds WhatsApp group size to ~99 participants for the message-plus-inbox write. Reaching for a graph DB (Neo4j) when your only queries are "who does A follow" and "who follows A" is a junior tell; graph DBs earn their keep only for multi-hop comprehensions.

**In-Memory Databases** -- Store data entirely in RAM for sub-millisecond access. Examples: Redis, Memcache. Used for caching, session storage, distributed locks, sorted sets (priority queues), geospatial indexes, Pub/Sub, atomic counters, rate-limit buckets. Volatile: data lost on restart unless persistence is configured. Redis is single-threaded but event-driven, capable of serving ~1M QPS for trivial ops from a single instance, but only ~100k ops/sec for heavier per-request work (the [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Rate Limiter]] budgets ~50k rate checks/sec per instance since each check is a read plus a write). [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] notes Redis' performance comes from event-driven architecture, not multi-threading.

**Blob Storage** -- For storing large, unstructured files (images, videos, PDFs, raw HTML, document snapshots). Services like Amazon S3 store blobs in buckets, each file getting a unique URL. Scalable, pay-as-you-go, automatic replication. The universal pattern: store opaque bytes in cheap blob storage and store queryable structured metadata in a database, linked by an S3 URL. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] stores file bytes in S3 and metadata in a DB; [[problems/WhatsApp/WhatsApp|WhatsApp]] stores media in S3, not DynamoDB (which is not built for hundreds-of-MB blobs); [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] stores raw HTML and parsed text in S3; [[problems/Google Docs/Google Docs|Google Docs]] stores large snapshots in S3 and keeps the pointer in Postgres. Profile pictures and media should store URIs pointing to S3/CDN, not binary in the database.

**Pre-Signed URLs** -- A time-limited, signed, constraint-bound URL (mime type, size, TTL on the order of minutes to an hour) that lets a client upload or download bytes directly to/from blob storage without routing them through your servers. The flow: client asks your service for an upload target; the service uses its credentials to request a pre-signed URL from S3; the client uploads straight to S3; the client then sends only the resulting URL through the normal path. Used by [[problems/WhatsApp/WhatsApp|WhatsApp]] for media (keeps the chat servers handling tiny control messages) and [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] for large files (no bytes touch your servers, and it sidesteps the API gateway body-size cap).

**Relational Schema Design** -- [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] teaches a methodology: list all features, identify entities and relationships, underline nouns (entities/attributes) and verbs (relationships/status changes). Key patterns: one-to-many (add foreign key in dependent entity), many-to-many (create mapping table), and knowing when to split entities even if they share attributes (education vs company tables, for extensibility). Storing lists as columns (e.g., skills as comma-separated) doesn't scale because lookups become full table scans.

**Time-Series / Analytics Databases** -- Column-oriented databases like ClickHouse or TimescaleDB are optimized for append-only, time-series data queried with aggregations (clicks per day, top referrers). Partition by date, drop old partitions for retention management. [[problems/URL Shortener/URL Shortener|URL Shortener]] uses ClickHouse for click analytics: each redirect pushes a click event to a message queue, consumed by an Analytics Service that writes to ClickHouse.

## Data Management & Performance

**Database Indexing** -- An efficient lookup structure that maps column values to row pointers, avoiding full table scans. Like a book's index. Create indexes on frequently queried columns (primary keys, foreign keys, WHERE conditions). Speeds up reads but slows writes (index updates on each write). Only index the most frequently accessed columns.

**Caching** -- Storing frequently accessed data in memory for faster retrieval than disk-based databases. The **cache-aside pattern**: check cache first (hit = instant return), on miss fetch from DB and store in cache. TTL (Time to Live) prevents serving stale data. Caching exists at multiple levels:
- **Browser cache**: stores HTML/CSS/JS locally, controlled by Cache-Control header
- **Server cache**: Redis or Memcache layer before the database
- **Database cache**: caching query results within or alongside the DB
- **CDN cache**: geographically distributed edge caching

[[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] uses the milk tea analogy: department store (hard disk) vs fridge (cache). Cache hit = milk in fridge. Cache miss = trip to store.

**Multi-Layer Caching** -- For ultra-low latency read paths, stack multiple cache layers. [[problems/URL Shortener/URL Shortener|URL Shortener]] demonstrates three layers: (1) Redis server-side cache for shortCode->longUrl lookups (80-90% hit rate given power-law distribution), (2) CDN edge caching for 30-60s (absorbs viral link spikes before they reach the API Gateway, but trade-off: cached redirects skip analytics), (3) browser caching via 301 redirects (zero server load for repeats, but kills analytics and prevents URL changes). Write-through on creation ensures the first redirect after creation is a cache hit.

**Cache Write Strategies** -- Write-around (bypass cache, write to DB directly), write-through (write to both simultaneously, ensures consistency), write-back (write to cache first, DB later, faster but risk of data loss on crash). [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] explains write-through as: every update goes through cache first, only returns success when both cache and DB are updated. Alternative: DB tells cache to invalidate stale entries on update.

**Cache Eviction Policies** -- LRU (Least Recently Used), FIFO (First In First Out), LFU (Least Frequently Used). Determine which items to remove when cache is full. [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]] uses LFU with a short TTL for a hot-post cache, since only the most viral posts need shielding and posts rarely change.

**Denormalization** -- Combining related or derived data so reads avoid joins or recomputation. Example: a `user_orders` table instead of separate `users` and `orders` tables, or a per-user precomputed-feed table holding copies of post IDs. Faster reads at the cost of increased storage and more complex writes (write amplification). Common in read-heavy applications. [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]] frames the precomputed feed as denormalized derived data traded for read speed.

**Database Replication** -- Creating copies of the database across multiple servers. One primary handles writes; multiple read replicas handle read queries. Data is copied from primary to replicas to stay in sync. Improves read performance and availability (a replica can become primary if the original fails). Approaches: master-slave (one writer, many readers) and master-master (multiple writers). The [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Rate Limiter]] uses async Redis replicas per shard (promote on primary failure), accepting minor data loss as consistent with its AP choice.

**Database Sharding** -- Splitting a database into smaller pieces (shards) distributed across multiple servers. Each shard holds a subset of data based on a sharding key (e.g., user ID, event ID, docId, client ID, URL). Reduces load per shard and speeds up queries. Also called horizontal partitioning. Strategies: range-based, directory-based, geographical, hash-based. Consider the shard key carefully: [[System Design Interview - Design Ticketmaster w a Ex-Meta Staff Engineer|Ticketmaster]] weighs event_id vs venue_id. For key-value access (URL shortener random codes), hash-based sharding distributes uniformly with no hot shard. [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Rate Limiter]] shards Redis buckets by client ID (you must route to the instance holding that client's bucket, never a random one). [[problems/Google Docs/Google Docs|Google Docs]] partitions the op log by docId so a doc's history stays local to one shard.

**Vertical Partitioning** -- Splitting a table by columns instead of rows. Useful when a table has many columns but queries only need a few. Improves query performance by reducing scanned data.

**Change Data Capture (CDC)** -- A pattern where changes to a primary data store are streamed to other systems. The classic form: the database write-ahead log (WAL) is read by a connector (Debezium), published to a stream (Kafka), consumed by workers that update downstream systems (e.g., Elasticsearch), keeping a search index in sync without coupling write logic. [[System Design Interview - Design Ticketmaster w a Ex-Meta Staff Engineer|Ticketmaster]] uses CDC to sync Postgres -> Elasticsearch. A related form: blob-store notifications. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] can use S3 object-creation notifications to update chunk status without trusting the client (caveat: S3 multipart upload does not fire per-chunk notifications, so the two do not compose).

**Trust but Verify** -- Never trust a client's success report when it can corrupt server state; have the server confirm with the source of truth before mutating. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]]'s preferred way to update chunk status: the client says "chunk uploaded," then the metadata service asks S3 to confirm the chunk actually exists before flipping status. The alternative (client PATCHes status directly) is insecure because a malicious client can lie and leave metadata inconsistent with S3.

## Scaling & Load Balancing

**Vertical Scaling (Scale Up)** -- Upgrading a single machine: more CPU, RAM, storage. Quick fix but limited: machines have max capacity, cost grows exponentially, single point of failure. Not a long-term solution for high traffic. [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] illustrates this with the Delicious.com story: founder kept buying better hardware until even the best wasn't enough.

**Horizontal Scaling (Scale Out)** -- Adding more machines to distribute workload. More capacity, better fault tolerance (one server down, others take over). Introduces the challenge of request routing, solved by load balancers. Scales linearly: the [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] math shows 1 machine = 10 days, so 2 = 5 days, and you provision ~4 for leeway.

**Load Balancing** -- Distributes incoming requests across multiple servers. Sits between clients and backend servers. If a server fails, the load balancer redirects traffic to healthy ones. Load balancers maintain a registry of all available servers with health status. Algorithms:
- **Round Robin**: sequential rotation, simple, works for uniform servers
- **Weighted Round Robin**: servers assigned weights by capacity, more powerful servers get proportionally more requests
- **Least Connections**: routes to server with fewest active connections, good for variable-length tasks and the key policy for persistent-connection servers (see Layer 4 below)
- **Weighted Least Connections**: combines least connections with capacity weights
- **Least Response Time**: combines fewest connections with lowest latency
- **IP Hashing**: consistent routing based on client IP, ensures session persistence
- **Geographical**: routes to closest server, reduces latency
- **Consistent Hashing**: uses a hash ring, ensures minimal redistribution when servers are added/removed

**Layer 4 vs Layer 7 Load Balancing** -- Layer 7 (application) load balancers terminate each request and route by headers/path/method to any stateless web server, richer but slower. Layer 4 (transport) load balancers route by IP/port, faster and more secure. The distinction is decisive for stateful persistent connections: [[problems/WhatsApp/WhatsApp|WhatsApp]] must use a **layer 4** load balancer for WebSockets because a layer 7 LB would terminate each request and forward to any server, but a WebSocket connection must persist to one specific server. A layer 4 LB creates a symmetric TCP connection (when the client connects, the LB opens a matching connection to a chat server; when the client disconnects, the LB disconnects too), making the LB effectively invisible. Pair it with a **least-connections** policy because the scarce resource is the open connection itself (~2M per chat server), so a freshly added server with near-zero connections naturally absorbs new inbound connections.

**Load Balancer Types** -- Hardware (F5 BIG-IP, Citrix), Software (HAProxy, Nginx), Cloud (AWS ELB, Azure LB, GCP LB). Load balancers are single points of failure; mitigate with redundant pairs (failover), monitoring, autoscaling, and DNS failover. [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] covers Layer 4 vs Layer 7 in depth. Load balancers also provide security by keeping application servers in a private network, and handle SSL/TLS termination.

**Connection Pooling** -- Keep a pool of persistent TCP connections to a downstream dependency and reuse them, eliminating the per-request TCP handshake cost (which can add 20-50ms depending on distance). The [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Rate Limiter]] relies on it for the Redis round trip in the request hot path; most Redis clients pool automatically, but you tune pool size to request volume.

**Geographic Co-Location** -- Place servers close to users and co-locate their dependencies in the same data center to minimize latency. The [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Rate Limiter]] puts gateways near users (a gateway in Tokyo for Tokyo users) with Redis in the same data center. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] imagines the whole setup replicated across ~5 data centers worldwide, which is part of why it can skip a CDN (users are usually near their own data).

**CDN (Content Delivery Network)** -- A global network of edge servers caching content close to users. Serves static assets (images, CSS, JS, video) from the nearest server, reducing latency. Pull-based CDN fetches on first request (good for regularly updated content). Push-based CDN requires manual upload (good for large, infrequently updated files). CDNs can also cache API responses for short periods (30s-1min), extremely effective for popular queries. Benefits: reduced latency, high availability, DDoS protection. CDN caching breaks down with highly personalized content or high query permutation counts. **CDN trade-off**: it is not always worth it. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] opts out, because the dominant pattern is users downloading their own files (already near their data center), and CDNs are expensive; it only helps travelers or a single super-popular shared file. Weighing whether a CDN is even needed is a top-candidate signal.

## Distributed Systems

**CAP Theorem** -- In a distributed system, when a network partition occurs, you must choose between Consistency and Availability. Partition tolerance is mandatory (network failures are inevitable). **CP systems** refuse requests they can't guarantee are correct (banking, airline reservations, ticket booking). **AP systems** always respond, possibly with stale data (social media feeds, live viewer counts, chat apps, file sync, rate limiters). The trade-off only kicks in during partitions; when healthy, you get both. The decision comes down to: what's the worse user consequence, wrong data or no service? [[CAP Theorem]] has detailed examples and a decision framework. Important nuance: different parts of a system can make different CAP choices. [[System Design Interview - Design Ticketmaster w a Ex-Meta Staff Engineer|Ticketmaster]] uses strong consistency for booking but high availability for search. [[problems/Uber/Uber|Uber]] uses strong consistency for ride matching but eventual consistency for location updates. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]], [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]], and the [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Rate Limiter]] all pick availability (stale reads beat blocked service). [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] teaches CAP through the "reminder service" analogy: husband and wife with diaries.

**PACELC Theorem** -- Extension of CAP: when there IS a partition, choose between Availability and Consistency. **Else** (no partition), choose between **Latency and Consistency**. Want consistency? Must wait for all nodes to sync (higher latency). Want low latency? Accept potentially stale data. CAP doesn't address latency, PACELC does. [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]]

**Eventual Consistency** -- Most AP systems eventually converge to the same state. When communication resumes between nodes, they exchange data and sync. There may be temporary inconsistency, but data converges over time. Good enough for social feeds, search suggestions, viewer counts, file sync, and presence. The convergence is often budgeted: [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]] gives a post 1 minute to propagate into followers' feeds, which is what licenses async fan-out. [[problems/Google Docs/Google Docs|Google Docs]] requires that clients may briefly diverge while ops are in flight but must converge to the identical document once all ops are delivered (convergence is non-negotiable, momentary divergence is fine).

**Consistent Hashing** -- A distribution scheme where adding/removing servers only relocates a minimal number of keys. Visualize a ring marked 0 to a large number. Servers and keys are hashed onto this ring. A key is assigned to the nearest server clockwise. **Virtual nodes** solve uneven distribution: each server gets multiple labels on the ring; more powerful servers get more labels, thus more keys. When a server is removed, only its keys redistribute to neighbors; all other keys stay put. Used in Kafka, Cassandra, sticky sessions, Redis Cluster (via hash slots, ~16,384 of them), routing stateful connections, and pinning work to a server. [[problems/WhatsApp/WhatsApp|WhatsApp]] uses it as one routing option (a user is owned by a specific server) and details its scaling cost: going 4 -> 5 servers remaps ~20% of users on the edge, so you drain misassigned users slowly to avoid a thundering herd, and during the window a sender may have to dual-send to both the old and new server. [[problems/Google Docs/Google Docs|Google Docs]] hashes docId onto a ring so every editor of a doc lands on the same collaboration server (keeping OT in-memory). [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] covers the math of data movement when changing server count.

**Distributed Locks** -- A mechanism for coordinating access to a shared resource across multiple service instances. Since horizontally scaled services run on many machines, they need a shared, external lock (not in-memory). Redis is commonly used: store a key with a TTL that auto-expires. [[System Design Interview - Design Ticketmaster w a Ex-Meta Staff Engineer|Ticketmaster]] uses a Redis lock (`ticket_id -> true, TTL: 10min`) for reservation expiry: no cron jobs, no reserved timestamps, the key simply disappears after 10 minutes. [[problems/Uber/Uber|Uber]] uses a Redis lock with `NX` (`SET lock:ride:{rideId} driver_id NX EX 30`): `NX` means "only set if absent," so the first driver to accept wins and all others fail atomically.

**Atomicity via Single-Threaded Redis & Lua Scripts** -- A read-then-write flow across many service instances has a read-after-write race: two gateways both read tokens=1, both accept, both write back, and the second overwrites the first, letting two requests pass on one token. Because Redis is single-threaded, you can make the whole read-compute-write sequence atomic by bundling it into one **Lua script** that goes over the wire once and runs to completion while others wait. The [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Distributed Rate Limiter]] uses a Lua script wrapping `HMGET` (read bucket fields), the refill computation, and `HSET` (write back) to kill the race; proactively spotting this race is flagged as a staff-level move. The Postgres equivalent is a transaction with locking.

**Idempotency** -- Ensuring that repeated identical requests produce the same result as a single request. Critical in distributed systems where retries are common (network failures, user refreshes). Implementation: assign each request a unique ID, check if already processed before executing. Prevents double-charges, duplicate orders, etc. [[problems/Twitter Feed/Twitter Feed|Twitter Feed]] uses a unique constraint on `(postId, userId)` in the Like table to enforce idempotent likes at the database level.

**Stateless vs Stateful Systems** -- **Stateless**: every server equally equipped to handle any request, no context stored. Easy to scale, resilient to failures, but state must be stored externally (DB), adding network IO latency. **Stateful**: context lives in the application server. Lower latency (no network IO), but not resilient to failures and harder to scale. Use stateful for short-lived state where latency is critical. The defining hardness of [[problems/WhatsApp/WhatsApp|WhatsApp]] is that its chat servers are stateful (they hold the live WebSocket and an in-memory clientId-to-socket map), which is also what makes routing and scaling hard. [[problems/Google Docs/Google Docs|Google Docs]] collaboration servers are stateful too (authoritative in-memory revision counter and recent op tail per doc). [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] covers the calculator (stateless) vs chatbot (stateful) examples and the PUBG match_ID routing problem.

**Fault Tolerance & Crash Recovery** -- A large system has constant component failures; no single failure can take down the app. A core technique with message queues: a pulled message is not truly removed until work is confirmed, so a crashed worker's message reappears for another. [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]]: in SQS the **visibility timeout** hides a pulled message; if the owner dies, the timeout elapses and the message becomes visible again, and on success the worker explicitly deletes it. In Kafka, a **consumer group** guarantees each message is read by one worker, which commits its offset only after storing to S3; messages age out by retention (default ~7 days) rather than on commit.

**Availability, SLOs, and SLAs** -- Availability is measured as uptime percentage. "Five nines" (99.999%) means ~5 minutes of downtime per year vs. 99.9% allowing ~8.76 hours. SLOs (Service Level Objectives) are internal performance goals (e.g., respond within 300ms 99.9% of the time). SLAs (Service Level Agreements) are formal contracts with customers, with penalties for breaches. Resilience is built through redundancy, graceful degradation, fault tolerance.

**Fail Open vs Fail Closed** -- When a protective component (like a rate limiter) loses its state store, you must pick a failure stance. **Fail open**: let every request through; risk is backends lose their protection and cascade-fail. **Fail closed**: reject everything; risk is the site appears down. The [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Distributed Rate Limiter]] leans fail closed for a protective limiter, but notes the smarter production middle ground: keep a single in-memory fixed-window counter per gateway as a temporary degraded mode while Redis recovers, accepting the loss of cross-gateway coordination so that some limiting survives.

**Throughput & Latency** -- Throughput measures volume over time: RPS (server), QPS (database), bytes/second (network). Latency measures time for a single request round-trip. Optimizing for one often sacrifices the other (batching increases throughput but adds latency). Quantified latency budgets recur as design constraints: 500 ms for [[problems/WhatsApp/WhatsApp|WhatsApp]] message delivery and [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]], ~100 ms for [[problems/Google Docs/Google Docs|Google Docs]] edit propagation, under 10 ms added by the [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Rate Limiter]] sitting in the hot path.

## Architecture Patterns

**Microservices** -- Breaking a monolithic application into smaller, independent services, each handling a single responsibility with its own database and logic. Services communicate via APIs or message queues. Each can scale and deploy independently. Trade-off: operational complexity increases (networking, monitoring, distributed debugging). Beware over-modularizing: [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]] rejects wrapping every table behind its own microservice API as over-engineering when the decoupling benefit is dwarfed by the overhead; do it only if the interviewer insists.

**Message Queues** -- Enable asynchronous communication between services. A producer places a message in the queue; a consumer retrieves and processes it. Decouples services, prevents overload, improves scalability. The consumer processes at its own pace regardless of producer throughput. If a consumer crashes mid-processing, the unacknowledged message returns to the queue for another instance (built-in fault tolerance). [[problems/Twitter Feed/Twitter Feed|Twitter Feed]] and [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]] use Kafka for feed fan-out. [[problems/Uber/Uber|Uber]] partitions a ride-request queue by geographic region to prevent head-of-line blocking (a hard-to-match remote ride does not block easy Manhattan matches). [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] uses queues as the frontier and between pipeline stages. **Kafka vs SQS** notables: Kafka has no built-in retries (implement via a separate fail topic with the next-run time in the message) and uses offsets + consumer groups + retention; SQS supports exponential-backoff retries and a dead-letter queue out of the box, and uses the visibility timeout for crash recovery.

**Pointer-on-Queue Pattern** -- Keep queue payloads tiny by putting a pointer (an S3 link) on the queue instead of the large blob, which lives in blob storage. SQS/Kafka default to ~1 MB max message size, and [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] pages average 2 MB, so the crawler puts `{ URL, S3 link }` on the parsing queue and the parser fetches the HTML from S3.

**Pipeline Staging** -- For any data-pipeline problem, split a monolithic worker into independent stages connected by queues. [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] splits the crawler (fetch HTML -> S3) from the parsing worker (parse text, extract URLs) via a parsing queue. Benefits: independent scaling per stage, observability (you can see where a URL is), retry isolation (the error-prone fetch is isolated), and robustness to changing requirements (if the ML team later wants OCR, you reload URLs into the parsing queue and rerun phase 2 against stored HTML rather than re-crawling the web).

**Retries with Exponential Backoff** -- Never retry on an in-memory timer (lost if the worker dies, and a fixed short wait rarely lets a struggling server recover). Persist the retry state: encode the next-run time in the message (Kafka fail topic) or use the SQS visibility timeout, which grows exponentially per attempt (30s, 2m, 5m, 15m). Cap retries with a max receive count, after which the message moves to a **dead-letter queue** (a queue for messages that cannot be processed). [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]]

**Jitter** -- Add randomness to retry/backoff waits to prevent a thundering herd. Without it, many workers rate-limited on the same domain re-enqueue, get pulled together, hit the limiter together, only one passes, and the herd repeats. [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] applies jitter to per-domain rate-limit re-enqueues. Related: drain remapped connections slowly during a consistent-hash rebalance for the same thundering-herd reason ([[problems/WhatsApp/WhatsApp|WhatsApp]]).

**Backpressure / Dynamic Downstream Scaling** -- Scale downstream stages off queue depth, since the bottleneck stage sets the pace. [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]]: fetching is the bottleneck, so parsing workers scale up automatically when the parsing queue backs up; downstream stages only need to keep up.

**Rate Limiting** -- Restricting the number of requests a client can make within a time frame. Prevents abuse, DDoS, and resource exhaustion. Standard production placement is at the **edge** (API gateway / load balancer), the bouncer-at-the-door model, so abusers are turned away before consuming backend resources; edge placement loses business context, so encode needed claims (premium status) in a JWT. Identify clients by user ID, IP, or API key, ideally with layered rules (authenticated > anonymous IP, premium > free). Algorithms (simplest to most capable):
- **Fixed window counter**: per-window count, trivial but suffers boundary bursts (200 requests across a window edge) and starvation.
- **Sliding window log**: exact timestamps in a rolling window, perfect accuracy but heavy memory (heap/deque).
- **Sliding window counter**: weighted blend of previous and current windows (e.g., current 6 + 30% of previous 8 = 8.4 vs a limit of 10), near-sliding accuracy with two integers, an even-distribution approximation.
- **Token bucket** (the usual pick): a per-client bucket where size = burst capacity and refill rate = sustained rate, one token per request; stores only token count + last refill timestamp. Two clean knobs separate bursts from sustained rate.

Reject over-limit requests **fail fast** with HTTP 429 (never queue interactive requests, which just grows a backlog). At scale, hold bucket state in Redis shared across all gateways, shard by client ID (a single Redis instance does ~50k rate checks/sec), make the check atomic with a Lua script, and use replicas for fault tolerance. The [[sources/Design a Distributed Rate Limiter w an Ex-Meta Staff Engineer|Distributed Rate Limiter]] is the full treatment; [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] applies a per-domain Redis sliding-window limiter (default 1 req/sec/domain) for politeness.

**Politeness (Crawler)** -- A web crawler must not effectively DOS a site. Honor **robots.txt** per domain: `user-agent` (which crawler the rules apply to), `disallow` (paths not to crawl), `crawl-delay` (seconds between requests). Fetch and cache robots.txt per domain (with a TTL since it changes), drop disallowed paths, and respect the crawl-delay (if `now - last_crawled < crawl_delay`, re-enqueue with the remaining delay as the visibility timeout). [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]]

**Fan-Out On Write (Push Model)** -- When a user posts, inject the post into all followers' pre-computed feed caches immediately. Feed reads become a simple cache lookup. Pros: blazing fast reads, decoupled from social-graph complexity at read time. Cons: write amplification (one post from a user with 10M followers = 10M writes). Best for the vast majority of users with moderate follower counts. Run the writes async via a worker pool, splitting large follower sets into sub-jobs. [[problems/Twitter Feed/Twitter Feed|Twitter Feed]], [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]]

**Fan-Out On Read (Pull Model)** -- Generate the feed at read time by querying all followees' posts. Pros: simple, always fresh, no extra storage. Cons: slow at read time (scales with followee count), painful for users who follow thousands. [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]]: a user following 10,000 people forces ~10,000 Post-GSI queries merging ~10 million posts on every load.

**Hybrid Fan-Out (Staff Answer)** -- Combine both. Precompute feeds for normal accounts (set a flag on the Follow edge); for celebrities above a follower threshold (~100k), do not fan out on write (flag off). At read time, read the precomputed feed AND separately pull the latest N posts of the not-precomputed (celebrity) follows live, then merge. This caps both write amplification (no broadcast for mega accounts) and read fan-out (only the handful of celebrities a user follows). This is what Twitter does. Also skip precomputing for inactive users. When a celebrity posts and that post lands in millions of feeds, watch for the resulting hot key (below). [[problems/Twitter Feed/Twitter Feed|Twitter Feed]], [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]], [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]]

**Hot Key / Hot Shard Problem** -- Skewed load on a single key (one viral post ID, one partition) throttles a single physical host regardless of total cluster capacity, because partitions map to physical hosts with real limits. [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]]: when a popular account posts, millions of feeds query the same post ID. Caching the hot partition (distributed cache, LFU, short TTL, only the most viral posts) is the common answer but incomplete, because a cache sharded by post ID inherits the same hot key. The deeper fix: **replicate the hot read instead of sharding it**. Run multiple independent, uncoordinated cache instances and have the reader pick one at random; now N requests reach the DB (N = number of cache instances) instead of millions. Recognizing that the cache itself inherits the hot key, and resolving it by replication + randomized reads, is the senior/staff signal.

**Two-Phase Booking** -- A pattern where a transaction is split into reserve and confirm steps. The user first reserves a resource (seat, item), gets a time window to complete payment. If the window expires, the reservation releases automatically. [[problems/Ticketmaster/Ticketmaster|Ticketmaster]] implements this with two endpoints: `POST /booking/reserve` (creates a Redis lock with TTL) and `PUT /booking/confirm` (processes payment via Stripe webhook, updates DB to booked). Security note: never put userId in the request body; use the JWT/session token in headers.

**Virtual Waiting Queue** -- A pattern for handling massive traffic surges (Taylor Swift tickets, Super Bowl). Instead of letting all users hit the system simultaneously, route them into a queue (Redis sorted set, priority by arrival time or random for fairness). Batch-release users as capacity opens up. Admin-enabled per event. Notify users via SSE when it's their turn. Protects backend services while improving UX. [[problems/Ticketmaster/Ticketmaster|Ticketmaster]]

**Async Counter Aggregation** -- A pattern for handling extreme write contention on a single row (e.g., like count on a viral post receiving 100K likes/minute). Instead of updating the Post row on every like, use Redis `INCR` on a counter key (`likes:{postId}`), atomic and handling 100K+ ops/sec. A background worker periodically flushes the Redis counter to the database (every 5-10s). The displayed count reads from Redis (real-time), the DB value lags slightly (eventual consistency, fine for AP). Reconcile with the Like table's actual count during flush. [[problems/Twitter Feed/Twitter Feed|Twitter Feed]]

**Trie (Prefix Tree)** -- A tree data structure for efficient prefix-based lookups. Each node represents a character, terminal nodes mark complete words and store frequency counts. Used for search autocomplete/type-ahead. Attach a **min-heap of size K** to every node to store the top K most popular completions, enabling O(1) query time. Reduce write contention by batching frequency updates in a hash map or by **sampling** (process every 100th request; random sampling still captures popular queries with 99%+ probability). Shard tries by prefix using consistent hashing. [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] covers this in the Google Search Type-Ahead mock interview.

**Inverted Index / Search** -- Text is tokenized into terms, and an inverted index maps each term to the documents containing it. Used by Elasticsearch and Twitter's Early Bird (built on Lucene). Supports geospatial queries (quadtrees + geohashing). Should NOT be used as a primary data store. Keep in sync with the primary DB via CDC or application-level dual writes. Twitter uses **scatter and gather**: a search query is distributed across global data-center shards, results merged and re-ranked.

**Elasticsearch & Search Optimization** -- A search-optimized database using inverted indexes. AWS OpenSearch (managed Elasticsearch) supports node query caching (LRU cache of top queries per shard). [[problems/Ticketmaster/Ticketmaster|Ticketmaster]] uses CDC to sync Postgres -> Elasticsearch for event search.

**Stripe Integration Pattern** -- Offload payment processing to Stripe. Post payment details via Stripe's client libraries. Stripe processes asynchronously and calls back via webhook when payment succeeds or fails. Your service exposes a webhook endpoint to receive the callback and update the order/ticket status accordingly.

## Real-Time Messaging & Connection Management

**Durable Delivery Decoupled from the Connection** -- Guaranteed delivery cannot rely on a live socket being up. [[problems/WhatsApp/WhatsApp|WhatsApp]] writes every send to durable storage independent of the connection: a **Messages table** (messageId, contents, senderId, server-assigned timestamp) plus an **Inbox table** (one row per recipient per undelivered message). The send is a single transaction (one Messages row + one Inbox row per participant). Connected clients get an immediate push; offline clients leave Inbox rows waiting. This split is what turns a transient "live chat room" into a system with a guarantee.

**Ack-Based Delivery Tracking & Reconnect Replay** -- On receiving a message the client sends an `ack` with the messageId; the server deletes that Inbox row so it is never resent. On reconnect, the server reads all the client's remaining Inbox rows, fetches the message contents, pushes them, and they are acked and removed. The guarantee: as long as a client periodically reconnects, every message is eventually delivered; if online, delivery is immediate. The same reconnect-and-replay idea appears in [[problems/Google Docs/Google Docs|Google Docs]] (client sends its last acknowledged revision and the server replays ops since then). [[problems/WhatsApp/WhatsApp|WhatsApp]]

**Cross-Server Connection Routing** -- Once persistent connections are spread across many stateful servers, a sender on server 1 cannot directly reach a recipient whose socket is on server 2. [[problems/WhatsApp/WhatsApp|WhatsApp]] compares three routes. (A) **Kafka topic per user**: the common instinct but wrong, Kafka is not built for billions of heavy (~50-100 KB) topics or rapid connect/disconnect of micro-topics. (B) **Consistent hash ring**: a user is owned by a specific server (drop the LB, expose servers via DNS, add a chat registry, store the user-to-server map in ZooKeeper/etcd); reaching B means looking up B's server and sending there (two hops), at the cost of scaling-window orchestration and dual-sends. (C) **Redis Pub/Sub** (recommended): each server subscribes to a Redis topic per connected user; to send, publish to the recipient's topic and Redis routes to whichever server holds that socket.

**Redis Pub/Sub as a Lightweight Relay** -- Redis Pub/Sub decouples real-time event publishers from the instance holding a given client's socket; it is a "dumb" socket relay that keeps an internal map of which sockets listen on a topic and forwards on publish. [[problems/Uber/Uber|Uber]] uses it for ride tracking (the Location Service publishes to `ride:{rideId}`, whichever Tracking Service holds the rider's WebSocket forwards it). [[problems/WhatsApp/WhatsApp|WhatsApp]] uses it for message routing across chat servers, accepting its **at-most-once** guarantee precisely because durability already lives in the Messages/Inbox layer (Redis gives the sub-500 ms real-time path; the durable store gives the guarantee). [[problems/Google Docs/Google Docs|Google Docs]] uses it for cross-server op fan-out during failover or when a doc is briefly split across instances. Cost at extreme scale: topics spread across cluster nodes mean each app server holds a connection to each Redis node (an n x m connection count).

**Presence / Online Status** -- Showing who is online. The hard part is not storing status, it is **fan-out**: one user connecting/disconnecting must not trigger tens of thousands of notifications. Two approaches from [[problems/WhatsApp/WhatsApp|WhatsApp]]: a **polling** status table (write available on connect, unavailable on disconnect, query on demand; cheap, eventually consistent) or a **push** approach reusing the existing Pub/Sub (users subscribe to the presence topics of users they care about). The real design move is to **constrain fan-out at the data model**: cap how many watchers a user can have, and degrade presence to lazy/polling for the long tail while pushing only for active conversations. Presence in [[problems/Google Docs/Google Docs|Google Docs]] is simply the membership of the connection set (broadcast join with color on open, leave on close or heartbeat timeout).

**Multi-Device Delivery (device, not user)** -- A person is not offline, a device is (your phone went dark but your laptop stayed on). Track delivery per device by introducing a **Client entity (userId + clientId)** and re-keying everything that was per-user to per-client: Inbox uses the recipient clientId, Pub/Sub topics become per-client, acks are per device (a message can be delivered-and-removed for the phone while still pending for the laptop). The cost is fan-out: 100 people x 5 devices is a large multiplier that collides with DynamoDB's 100-record transaction limit, mitigated by capping devices per user (~2-3) and percolating that into the allowed group size. [[problems/WhatsApp/WhatsApp|WhatsApp]]

**Toxic-Sludge Retention (TTL cleanup)** -- Retained private data is a liability, so keep messages only as long as needed to deliver them. [[problems/WhatsApp/WhatsApp|WhatsApp]] holds an undelivered message ~30 days then drops it, via a secondary index on the Messages timestamp (and a timestamp on Inbox to sweep stragglers); when the last Inbox row for a message is deleted (all clients received it), delete the Messages row too. This time-based cleanup is what keeps the live store at a few hundred TB rather than unbounded.

## Collaborative Editing

**Operations vs Whole-Document Saves** -- The canonical wrong answer for a collaborative editor is last-write-wins on a whole-document blob: two users who both load revision 5 and `PUT` the entire body silently destroy each other's work, and there is no real-time element. The fix is to send **fine-grained operations** (insert/delete/format at a position) instead of the whole document, and reconcile concurrent ops instead of overwriting. Each op is tagged with the `baseRevision` it was authored against, so the server knows which concurrent ops to reconcile it with. Local edits apply **optimistically** (instantly) for zero perceived latency, then sync. [[problems/Google Docs/Google Docs|Google Docs]]

**Operational Transformation (OT)** -- Ops are expressed against positions; when two were authored concurrently (same baseRevision), a **transform function** rewrites one so it composes correctly after the other (if A's position is left of B's, A is unchanged; if to the right, shift A by B's length delta; if equal, break the tie deterministically by authorId). Example: both insert at position 3 of "cat"; the server applies Alice's "s" (revision 6), then transforms Bob's op against it so his insert shifts to position 4, yielding "cats!" for everyone. The transform must satisfy the **TP1 property** (applying two concurrent ops in either order yields the identical document), which guarantees convergence. Strengths: tiny ops, a plain string representation, small memory. Weaknesses: correctness is famously hard (every op-type pair must be handled), and OT relies on a central server to assign canonical order, so it is not designed for peer-to-peer. [[problems/Google Docs/Google Docs|Google Docs]] chooses OT because a central collaboration server already exists, making OT's central-authority requirement free.

**CRDTs (Conflict-free Replicated Data Types)** -- Instead of transforming positions, give every character a globally unique immutable identity and a position from a total order that never needs rewriting. Sequence CRDTs (RGA, LSEQ, the models behind Yjs and Automerge) assign each inserted character a dense/fractional position id between its neighbors, so concurrent inserts at the same spot get distinct stable ids; deletes become tombstones (marked dead, never reused). Strengths: order-independent by construction, merges are commutative and idempotent, no central transform server, the natural fit for offline-first and true P2P. Weaknesses: heavy metadata (every character carries an id, deleted characters linger as tombstones), and tombstone garbage collection across replicas is its own hard problem. The [[problems/Google Docs/Google Docs|Google Docs]] decision: OT for the central-server case, CRDT named as the alternative for serverless P2P or robust offline merge, and the explicit trigger to reconsider CRDT if long offline windows became a hard requirement.

**Per-Document Collaboration Server** -- Co-locate all editors of a given doc on the same server process, which holds the authoritative in-memory state for that doc (current revision counter, recent op tail for transforming late ops, the live connection set). This is what makes OT cheap: every client transforms only against the linear server-assigned sequence, not an arbitrary peer mesh. Route editors to the right server by **consistent hashing on docId**, with **Redis Pub/Sub on `doc:{docId}`** as the cross-server fan-out fallback during rebalancing or failover. [[problems/Google Docs/Google Docs|Google Docs]]

**Event Sourcing & Op Log** -- The document's true state is the ordered fold of all its ops, appended to an Operation table keyed by `(docId, revision)` with a unique constraint as the integrity backbone (no two ops claim the same slot). Append-only writes mean no update contention, and ops for one doc are serialized through one collaboration server so the DB never sees a true write race. This gives durability for free (the op is persisted before ack), is the source material for OT transforms, and makes reconnect-replay trivial ("give me all ops after revision N"). The same append-only-log-plus-cursor idea is the [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] event-bus sync option (replay from a cursor, retains history for audit/version/recovery). [[problems/Google Docs/Google Docs|Google Docs]]

**Snapshots & Compaction** -- Pure op replay does not scale (a doc edited for a year has millions of ops). Periodically (e.g., every 1,000 ops) a background job folds the op log up to revision S into a full materialized body stored as a blob in S3 with a pointer in a Snapshot table. A late-joining client then loads the latest snapshot and replays only the ops with `revision > S` (at most ~1,000), bounding load work regardless of document age. Once a snapshot at S is durable, older ops are needed only for version history and can be compacted away or moved to cold storage. [[problems/Google Docs/Google Docs|Google Docs]]

**Offline Edit Reconciliation as OT Rebase** -- A user editing offline queues ops against the last revision they saw (say 40) while the server advances (to 70). On reconnect, the client pulls ops 41-70 and transforms its queued ops against that batch using the same OT transform function, then submits them for revisions 71+. Offline reconciliation is the same machinery as live concurrency, just with a longer gap. This is also the scenario where a CRDT's merge-anywhere property would shine. [[problems/Google Docs/Google Docs|Google Docs]]

## File Sync & Large Files

**Chunking** -- Split a large file into fixed-size pieces (~5 MB) on the client to bypass request-body limits, parallelize transfer (multiple chunks in flight, adaptive size based on network), and enable resume. A 50 GB file at ~100 Mbps takes over an hour, so restarting on a dropped connection is unacceptable. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]]

**Resumable Uploads** -- Track per-chunk status (id, status, s3Link) so a failed transfer re-uploads only the missing chunks. Resume flow: re-fetch metadata, compare the client's chunks to the stored chunks, upload only what is missing. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]]

**Content-Addressable Storage / Fingerprinting** -- Hash a chunk's bytes to get a stable unique ID, then use the fingerprint as the chunk's identity. This makes "this client chunk equals that stored chunk" reliable (naive index-tracking is error-prone), enabling deduplication and trustworthy comparison of client vs server state. To resume, compare client fingerprints against stored fingerprints and re-upload any not marked uploaded. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]]

**Multipart Upload** -- S3's built-in primitive that does client chunking, upload, fingerprinting, and validation for you. If you use it you would not also use S3 per-chunk notifications (multipart does not fire them). [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]]

**OS File-Watch APIs** -- Detect local folder changes natively: `FileSystemWatcher` on Windows, `FSEvents` on macOS. The sync client watches the local folder and uploads on a detected change. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]]

**Adaptive Polling** -- Vary poll frequency by activity (increase when the app is open or there is recent folder activity, back off otherwise) instead of a fixed interval, and a manual refresh button covers the impatient user. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] deliberately rejects WebSockets and long polling here: a persistent socket per client plus a manager is unjustified overhead when the product only needs updates within seconds, not milliseconds.

**Delta Sync** -- When a file changes, transfer only the chunks that changed (tracked by an updatedAt per chunk), not the whole file, especially a 50 GB one. A brand-new file downloads fully; an edit downloads only the changed chunks and the client stitches them back together. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]]

**Event Bus with Cursor (sync option)** -- Put every change event on a bus (Kafka); each folder has a sync cursor marking the last event read. First sync replays all events to build local state; each poll navigates from the cursor and applies only new events, then advances. Powerful because it retains history (audit trail, version control, recovery, roll back/forward), but overkill if versioning is not a requirement, in which case [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] polls the database directly for simplicity. The decision hinges on whether history/versioning is in scope.

**Reconciliation (integrity backstop)** -- Despite best efforts, local and remote can diverge (a bug, a dropped connection). Periodically (daily/weekly) the client fetches all remote info for the folder and compares it by fingerprint against the local DB and the files on disk, fixing any mismatch. Real-time-ish sync does the day-to-day work; reconciliation guarantees eventual integrity. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]]

**Compression Trade-Off** -- Compress on the client before upload to transfer fewer bytes, but only when worth it: text/DOCX compress very well, already-compressed media (JPEG, PNG, MP4) only shrinks a few percent and is not worth the CPU. Do intelligent, conditional compression by file type and network conditions, and store a `compressionAlgo` field in metadata so the other side knows how to decompress. [[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]]

## Crawling & Deduplication

**Frontier Queue** -- The set of URLs yet to be crawled, implemented as a message queue (SQS or Kafka), seeded with the seed URLs and drained as the crawl loop runs (take a URL, resolve DNS, fetch HTML, extract text and store it, extract URLs back onto the frontier, repeat until empty). [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]]

**Smart Scheduler (per-domain backlog)** -- Extracting URLs from one page tends to yield many URLs from the same domain, which then all bounce off the crawl-delay or rate limiter, wasting cycles. Instead of pushing extracted URLs straight onto the frontier, write them into URL metadata (with crawled state and last-crawled time); when the frontier runs low, a scheduler queries the metadata and applies an ordering algorithm (or priority queue) to interleave domains, avoiding hammering one. Also the basis for continual re-crawling (re-enqueue stale URLs by last-crawled time). [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]]

**URL Deduplication** -- Extracted URLs are heavily duplicated. Make the URL the primary key in URL metadata and check existence on insert; to avoid enqueuing duplicates, the parsing worker writes a new URL into metadata at enqueue time (last-crawled = undefined) and fills in real values once parsed. At 10 billion rows, shard on the URL; lookups stay O(log n). [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]]

**Content Deduplication via Hashing** -- Two different URLs (even on different domains) can have identical content, more often than expected. Hash the HTML in the crawler, store the hash in URL metadata, and before pushing to the parsing queue check whether that content hash already exists; if so, skip. Efficient membership check options: a DynamoDB **global secondary index** on the hash (O(log n), colocated, no extra hardware, the presenter's pick) or a Redis **hash set** (O(1); ~200 GB fits one box for 10^10 hashes, but extra hardware and persistence to maintain). [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]]

**Bloom Filter** -- A space-efficient probabilistic set-membership structure: returns "possibly in set" or "definitely not in set" (false positives possible, false negatives not). Trades accuracy for space. For content dedup a false positive means silently dropping real content. The [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] pushes back hard on the reflexive reach for a bloom filter (from the popular prep books): justify the memory constraint first. Since 200 GB of hashes fits a single Redis box, the bloom filter is the wrong default here; only use it if explicitly told the store must be tiny.

**Crawler Traps & Max Depth** -- Pages designed to keep crawlers on a site indefinitely (a page linking to itself, or to endless near-empty pages). Bound recursion with a per-URL `depth` field: when the parsing worker adds a newly extracted URL, set its depth to the parent's depth + 1; if depth exceeds a threshold (e.g., 20), do not enqueue it. [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]]

## Geospatial Systems

**Geospatial Indexing** -- Standard B-tree indexes are optimized for one-dimensional data and perform poorly on two-dimensional lat/lng queries. Geospatial indexes solve this. Two main approaches:

**Quad Trees** -- Recursively split a map into 4 regions; each region is a tree node. A threshold K determines when to split further (more than K entities -> split). Good for **uneven density** (dense NYC, empty ocean). Bad for high-frequency writes because adding/removing entities requires re-indexing the tree. PostGIS supports quad trees. [[Design Uber w a Ex-Meta Staff Engineer - System Design Interview breakdown|Uber interview]]

**Geohashing** -- Recursively split the map into 4 regions but density-independent: split until reaching desired precision, producing a base-32 string (longer = more precise). Easy to calculate and store (just a string), no re-indexing on writes. Redis supports it natively via `GEOADD` and `GEOSEARCH`. Less good for uneven distribution but excellent for high-frequency writes. [[Design Uber w a Ex-Meta Staff Engineer - System Design Interview breakdown|Uber interview]]

**Geospatial Decision Framework** -- Quad tree for read-heavy with uneven density (Yelp, Google Maps POIs); geohash for write-heavy with frequent updates (Uber driver locations, 600K updates/sec). Uber actually uses H3 (hexagonal grid) in production, which fixes the square's center-to-corner vs center-to-edge distance discrepancy, but quad trees + geohashing are sufficient for interviews.

**Redis Geospatial for Location Tracking** -- [[problems/Uber/Uber|Uber]] uses `GEOADD drivers:{city} lng lat driver_id` for writes and `GEOSEARCH drivers:{city} FROMLONLAT lng lat BYRADIUS 3 km ASC COUNT 20` for proximity queries, O(log N) for both. Partition by city/region for smaller working sets and regional data-center placement. Ephemeral data: if Redis crashes, drivers resend location within ~5 seconds, so zero practical loss. **Adaptive updates** cut the 600K TPS: the client sends conditionally (not accepting rides -> don't update; parked 20 minutes -> reduce frequency; remote area -> lower precision), down to 100K or less.

## URL Shortening

**URL Shortening** -- [[problems/URL Shortener/URL Shortener|URL Shortener]]: use base-62 encoding (a-z, A-Z, 0-9) for short codes. 62^7 gives ~3.5 trillion unique URLs, enough for centuries. Pre-generate keys via a **Key Generation Service (KGS)** to avoid collisions: a background process creates millions of random 7-char keys stored with `used = false`. Each URL Service instance pre-fetches a batch (e.g., 1,000 keys) into memory; if it crashes, those ~1,000 keys are lost (negligible out of trillions). Use 302 redirects (not 301) to preserve analytics: 301 tells the browser to cache the redirect permanently, so repeat clicks never hit the server. Cache shortCode->longUrl in Redis for ultra-low-latency reads. Push click events to a message queue for async analytics in ClickHouse. [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] adds: don't use auto-increment IDs (guessable), don't use UUID/MD5 (too long, collision-prone as the pool depletes).

## Computer Architecture Basics

**Storage Hierarchy** -- Disk (HDD/SSD, non-volatile, TB-scale, slowest), RAM (volatile, GB-scale, ~5000 MB/s), Cache (L1/L2/L3, MB-scale, nanosecond access), CPU (fetches/decodes/executes instructions from compiled machine code). The CPU checks L1 -> L2 -> L3 -> RAM in order. Motherboard connects all components.

**Production App Architecture** -- CI/CD pipelines (Jenkins, GitHub Actions) automate deployment. Load balancers/reverse proxies (Nginx) distribute traffic. External storage servers hold data. Logging/monitoring systems (PM2 backend, Sentry frontend) track issues. Alerting integrates with Slack. Debugging follows: identify -> replicate in staging -> debug -> hotfix. Observability also benefits from pipeline staging (you can see where work sits per phase, [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]]).

## System Design Interview Framework

**5-Step Roadmap** -- Requirements -> Core Entities -> APIs -> High-Level Design -> Deep Dives. The first 3 steps take ~15 minutes. High-level design satisfies functional requirements (build the simplest single-node thing first, deliberately, then evolve). Deep dives satisfy non-functional requirements. [[System Design Full Course - Software Architecture - Caching - CAP Theorem @SCALER|SCALER course]] presents a similar framework: Gather Requirements -> Estimate Scale -> Design Goals -> Design for Single Server -> Scale for Numbers.

**Functional Requirements** -- Features of the system ("users should be able to..."), derived by putting yourself in the end user's shoes and walking the flow. Keep to 3-5 core features. Do not waste time enumerating below-the-line items you will not build; ask the interviewer if they have anything in mind, then move on (over-expanding scope creates time problems later, per [[sources/Design WhatsApp w an Ex-Meta Senior Manager|WhatsApp]] and [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]]).

**Non-Functional Requirements** -- The qualities that make the system challenging, ideally quantified, and each one should map to a later deep dive. Don't list generic "-ilities." Put each quality in context and attach a number (a 500 ms latency budget, a 1-minute propagation budget, 50 GB max file size, under 10 ms added latency). Identify the CAP trade-off and the read/write profile specific to this system. Note out-of-scope items and check in with the interviewer.

**Scale Estimation (estimate to decide, not to impress)** -- Start with daily active users, derive storage and TPS. The senior habit: do back-of-envelope math only at a junction where the number changes a design decision ("Do I need to shard?", "How many crawler machines?", "Can Postgres handle this TPS?"), and overcommunicate that you are deferring math until then. Math just to conclude "that's a lot" teaches nobody anything. [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] does it exactly once, to compute machine count (~25k pages/sec/machine at 100% bandwidth, ~30% usable, so 10^10 pages / 10^4 per sec ~ 10 days on one machine, hence ~4 machines). [[Design Uber w a Ex-Meta Staff Engineer - System Design Interview breakdown|Uber interview]]: tell the interviewer upfront you'll estimate during the design when it influences decisions; 99.9% say yes.

**Core Entities** -- The data persisted and exchanged. List them early but don't detail fields until the high-level design evolves them naturally. Getting them wrong cascades into a wrong data model and API. Watch for the entity that unlocks the hard part later: the Client/Device entity in [[problems/WhatsApp/WhatsApp|WhatsApp]] (delivery is per device, not per user), the Operation/op-log entity in [[problems/Google Docs/Google Docs|Google Docs]], the Domain entity in [[sources/Design a Web Crawler w an Ex-Meta Staff Engineer|Web Crawler]] (robots.txt is per domain).

**APIs** -- User-facing REST endpoints, one per functional requirement (the safe default), using core entities as inputs/outputs. For real-time bidirectional systems use a WebSocket command model instead; for internal components describe a function/RPC interface or inputs/outputs. Use cursor-based pagination for feeds. First-pass endpoints are often intentionally wrong and get corrected after the deep dives ([[sources/Design Dropbox or Google Drive w an Ex-Meta Staff Engineer|Dropbox]] uploads stop posting bytes through the server once pre-signed URLs and chunking arrive).

**Database Choice** -- Focus on required qualities (ACID, write throughput, keyed lookups, ordered appends, query patterns) rather than the SQL vs NoSQL debate, which several sources call outdated. Pick what you know and justify briefly. Always start with RDBMS in interviews.

**Communicate Knowingly-Simple Decisions** -- High-level design and APIs are iterative; build the simplest single-node version first and keep flagging where you are knowingly simplifying so the interviewer does not assume you missed it. A silent obvious omission scores against you; proactively naming a gap protects the score. The process (find a bottleneck, propose a fix, find the bottleneck in that fix, repeat) is much of the hiring signal and is often never written on the whiteboard. [[sources/Design WhatsApp w an Ex-Meta Senior Manager|WhatsApp]], [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]]

**Level Expectations** -- A consistent rubric across sources. **Mid-level**: ~80% breadth, 20% depth; reach a solid high-level design and competently answer follow-ups; basic solutions (a cron job, an in-memory note that it won't scale) can pass; not expected to be perfect out of the gate. **Senior**: proactively drive, go genuinely deep in one or two places, state problems before being asked (a single Redis node is not enough, here's the math and shard count), discuss failure modes. **Staff**: reach the high-level design fast, work confidently through every NFR, show real depth in about three places, and zoom in on the qualitatively interesting parts (that chat servers are stateful, that the data model must serve fast reads and fast writes); staff are not "better seniors," they focus on different pieces with a larger toolkit. Never monopolize; leave room for interviewer steering. Managers are usually held to a senior standard. Interviewers commonly take control and drill a specific weak point even after you acknowledged it, to calibrate depth, not because you got it wrong.

**Knowing When to Stop** -- Favor simple, tightly-fit designs early and layer complexity only when needed; reaching for a graph DB, a bloom filter, or a CDN before establishing the need is a junior tell. Many platforms enforce product limits rather than engineer to infinity (Google won't page you to result 1000, LinkedIn caps connections, WhatsApp caps group size). It is often not worth engineering around the 0.001% of users; at some point you throw in the towel. [[sources/Design FB News Feed w an Ex-Meta Senior Manager|FB News Feed]]

**Interview Red Flags** -- Jumping directly to NoSQL without justification. Naming technologies (Redis, Elasticsearch, a graph DB, a bloom filter) without explaining the architectural decision or establishing the constraint. Not asking clarifying questions ("what am I rate limiting?", "dynamic JS content?"). Spending too long on scale estimation. Reaching for a heavy tool reflexively from a prep book.
